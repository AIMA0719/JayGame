JayGame UI 리디자인 - 접근 방식 선택

현재 C++ OpenGL 렌더링의 픽셀 문제를 해결하고, 레퍼런스(운빨존많겜, S2RD) 수준의 UI를 구현하기 위한 방향

현재 문제점

  • GL_NEAREST 필터링 — 텍스처가 확대될 때 픽셀이 그대로 보임
  • 5x7 비트맵 폰트 — TextRenderer가 프로시저럴 도트 폰트 사용, 레퍼런스 게임과 큰 차이
  • UISystem이 기본 색상 사각형 — 버튼/패널이 단색 직사각형으로 렌더링
  • 64x64 스프라이트 — 저해상도 아틀라스, 확대 시 깨짐
  • 배틀 HUD가 C++ 자체 렌더링 — 운빨존많겜처럼 세련된 UI 구현이 어려움
A

C++ 렌더링 개선 (현재 구조 유지)

방법: GL_LINEAR 필터링, 고해상도 아틀라스(128x128+), 9-patch UI 시스템, TTF 폰트 렌더링(FreeType) 추가

장점

  • 기존 코드 구조 유지
  • 배틀 중 성능 최적화 용이
  • 단일 렌더링 파이프라인

단점

  • UI 개발 속도가 매우 느림
  • 운빨존많겜 수준 UI 구현이 현실적으로 매우 어려움
  • FreeType, 9-patch 등 추가 라이브러리 필요
  • 레이아웃/애니메이션 직접 구현 필요
B

하이브리드: 배틀=C++ + UI=Compose 오버레이 (추천)

방법: 배틀 로직/유닛/적/투사체는 C++에서 렌더링, HUD/버튼/패널/정보는 Compose View 오버레이로 처리. 메뉴 화면은 이미 Compose.

장점

  • 배틀 성능 유지 (C++ 게임 루프)
  • UI는 Compose로 빠르게 레퍼런스 수준 구현 가능
  • 이미 MainActivity에 ComposeView 오버레이 기반 있음
  • Material Design, 커스텀 폰트, 애니메이션 즉시 사용

단점

  • C++↔Compose 간 실시간 통신 필요 (JNI 콜백)
  • 오버레이 레이아웃 조정 필요
C

전면 Compose 전환

방법: 배틀 화면까지 포함해서 전부 Compose Canvas + 커스텀 드로잉으로 전환

장점

  • 단일 기술 스택 (Kotlin only)
  • UI 일관성 최고

단점

  • C++ 엔진 코드 전체 재작성 (수천 줄)
  • 60Hz 게임 루프 성능 보장 어려움
  • 오브젝트 풀링, 공간 해싱 등 재구현 필요
  • 가장 시간 소모가 큼
